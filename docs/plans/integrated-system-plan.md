# AI 기반 배터리 장비 관리 시스템 통합 계획서

**작성일:** 2026-05-15  
**버전:** 1.0  
**목적:** 개발 진행 현황 공유 및 향후 방향 정의

---

## 1. 시스템 목표

> 배터리 충방전 검사 장비를 원격 제어하고, 이상 발생 시 AI가 자동으로 원인을 분석하여  
> 운영자가 빠르게 판단·조치할 수 있는 통합 관리 시스템 구축

```
장비 → 이상 감지 → AI 자동 분석 → 운영자 조치 → 재실행 승인 → 장비 재제어
                                   ↑
                        과거 이슈 이력(RAG) 기반 답변
```

## 3. 개발 진행 단계

### Phase 1 — 지식 기반 구축 ✅ 완료

**목표:** 과거 이슈/알람 이력을 AI가 검색할 수 있는 지식 베이스 구축

| 기능 | 구현 내용 |
|------|-----------|
| 문서 인덱싱 | BUG-*, INCIDENT-*, BATTERY-* Markdown → ChromaDB |
| RAG 검색 | 코사인 유사도 기반 관련 청크 추출 |
| AI 답변 생성 | Claude + 검색 컨텍스트 → 증상/원인/조치 4섹션 답변 |
| LLM 백엔드 이중화 | `LLM_BACKEND=claude` (인터넷) / `ollama` (폐쇄망) |

```
이슈 문서 (MD) → FastEmbed 임베딩 → ChromaDB 저장
질문 입력 → 유사 청크 검색 → Claude 답변 생성
```

**핵심 API**
- `POST /api/v1/index` — 문서 인덱싱
- `POST /api/v1/query` — RAG 질문 답변
- `POST /api/v1/search` — 유사 문서 검색만

---

### Phase 2 — 배터리 알람 자동 QA ✅ 완료

**목표:** 장비 알람 발생 시 AI가 자동으로 QA 리포트 생성

| 기능 | 구현 내용 |
|------|-----------|
| 알람 수신 API | `POST /api/v1/alarm/ingest` |
| Stage 1 — 구체화 | 알람 페이로드 → RAG 기반 이슈 상세화, 심각도 추정 |
| Stage 2 — 테스트 가능성 | `validation_criteria.yaml` 기준 판정 |
| Stage 3 — 리포트 | `data/qa_reports/QA_BATTERY_*.md` 자동 저장 |
| 알람 어댑터 | `alarm_adapter.py` — 장비 API 폴링 / 시리얼 수신 |

```
장비 알람 → alarm_adapter.py → POST /alarm/ingest
                                      │
                            Stage 1: 이슈 구체화 (RAG)
                            Stage 2: 테스트 가능성 판정
                            Stage 3: QA 리포트 자동 저장
```

**심각도별 처리 기준 (validation_criteria.yaml)**

| 심각도 | 자동화 필수 | 재현 단계 | 처리 방법 |
|--------|------------|---------|----------|
| Critical | ✓ | 1단계 이상 | 즉각 알림 + 자동 리포트 |
| High | - | 2단계 이상 | 자동 리포트 |
| Medium | - | 제한 없음 | 자동 리포트 |
| Low | - | 제한 없음 | 로그만 |

---

### Phase 3 — 대화형 챗봇 ✅ 완료

**목표:** 운영자가 자연어로 이슈를 질문하고 AI가 답변

| 기능 | 구현 내용 |
|------|-----------|
| 채팅 세션 관리 | PostgreSQL 기반 대화 이력 저장 |
| 컨텍스트 유지 | 이전 대화 내용을 AI에 전달 |
| 미답변 질문 추적 | RAG 검색 결과 없는 질문 자동 기록 |
| 이슈 제출 | 사용자가 직접 이슈 내용 제보 |
| 웹 UI | `/chat` 대화 인터페이스 |

---

### Phase 4 — 장비 원격 제어 ✅ 완료 (2026-05-15)

**목표:** 중앙 서버에서 다수의 장비를 원격 제어하고, 이상 발생 시 AI가 실시간 분석

#### 4-1. 장비 제어 기반

| 기능 | 구현 내용 |
|------|-----------|
| 장비 등록/관리 | IP·포트·유형 등록, 상태 관리 |
| 시퀀스 실행 | 다단계 명령 (충전→측정→방전) 순서 실행 |
| 하트비트 | 에이전트가 5초마다 상태·센서 데이터 전송 |
| E-STOP | 전체 장비 즉시 비상 정지 |

#### 4-2. 사용자 승인 플로우

```
운영자 [실행 버튼]
        │
        ▼
  pending_approval  ← 승인 대기 (대시보드에 표시)
      │       │
   [승인]   [거부]
      │       │
  running  cancelled
      │
   done / error
```

**도입 이유:** 오조작 방지, 검사 조건 최종 확인 후 실행

#### 4-3. 이상 감지 → AI 자동 분석

```
하트비트 수신 (5초)
      │
      ▼
  임계값 초과?  ──No──→ 정상
      │ Yes
      ▼
anomaly_logs 기록
      │
      ▼ (백그라운드)
  RAG 분석 질문 생성
  "장비 MOCK-01에서 temperature 이상. 측정값 52.5°C (임계값 45°C)"
      │
      ▼
  IssuePipeline.query() → 과거 이력 검색 → AI 답변
      │
      ▼
  anomaly_logs.rag_analysis 업데이트
  대시보드 이상 감지 패널에 표시
```

**임계값 (10분 쿨다운)**

| 항목 | Warning | Critical |
|------|---------|----------|
| 온도 | 45 °C | 55 °C |
| 전압 | 4.25 V | 4.35 V |
| 전류 | 3.0 A | 4.0 A |

#### 4-4. 장비 에이전트 (C#)

| 구성 | 설명 |
|------|------|
| `IHardwareDriver` | 하드웨어 추상 인터페이스 |
| `MockDriver` | 하드웨어 없이 소프트웨어 시뮬레이션 |
| `ModbusTcpDriver` | EasyModbusTCP 기반 실 장비 연결 |
| `HeartbeatBackgroundService` | 5초 주기 상태 전송 + 임계값 체크 |

**C# 선택 이유:** 팀 디버깅 환경에 적합, 기존 .NET 생태계 활용 가능

## 4. 3가지 핵심 동작 플로우

### 플로우 A — 정상 제어 흐름

```
운영자 → 시퀀스 생성 → 실행 요청 → [승인]
  → 에이전트에 명령 전송 → Modbus로 장비 제어
  → 완료 콜백 → 다음 스텝 → ... → 시퀀스 완료
```

### 플로우 B — 이상 감지 흐름 (AI 연계)

```
장비 이상 → 에이전트 하트비트 → 중앙 서버 임계값 체크
  → anomaly_logs 기록
  → RAG 분석 (백그라운드): 과거 유사 알람 이력 검색
  → AI 답변 생성: 예상 원인 + 즉각 조치 방법
  → 대시보드 이상 감지 패널 표시
  → 운영자 확인 → (필요 시) E-STOP
```

### 플로우 C — 운영자 챗봇 질문

```
운영자: "지난주 OVP-001 알람 원인이 뭐야?"
  → RAG 검색: ALARM-*, BATTERY-* 이력 문서
  → AI 답변: 과거 유사 사례 기반 원인 분석 + 조치 방법
  → 대화 이력 저장 (재조회 가능)
```

## 7. 향후 개발 항목 (Phase 5)

### 단기 우선순위

| 항목 | 내용 | 연계 |
|------|------|------|
| 이상 발생 시 시퀀스 자동 일시정지 | Critical 이상 감지 시 현재 시퀀스 PAUSED로 전환 | Phase 4 → Phase 3 |
| 이상 분석 결과를 챗봇 세션으로 연결 | anomaly_logs의 RAG 분석 결과 → 채팅 세션 자동 생성 | Phase 4 → Phase 3 |
| 검사 완료 후 자동 QA 리포트 | 시퀀스 DONE 시 Stage 3 리포트 자동 생성 | Phase 4 → Phase 2 |

### 중기

| 항목 | 내용 |
|------|------|
| 실 Modbus 드라이버 검증 | 실제 장비 레지스터 맵 확인 후 `ModbusTcpDriver.cs` 보정 |
| 다중 장비 스케일 테스트 | 장비 PC 5대 이상 동시 제어 검증 |
| 이상 이력 트렌드 분석 | anomaly_logs 누적 데이터 기반 주간 리포트 |
| 승인 권한 관리 | 역할별(운영자/관리자) 승인 권한 구분 |

### 장기

| 항목 | 내용 |
|------|------|
| 예측 분석 | 과거 이상 패턴으로 사전 경고 모델 |
| 자동 보고서 | 1일 1회 장비 상태 + 이상 이력 PDF |
| 모바일 알림 | 이상 감지 시 Slack/Email 즉시 알림 |

## 8. 다음 단계 연결 포인트 (Phase 5 설계 방향)

Phase 4까지 구현된 이상 감지 + RAG 분석이 Phase 3 챗봇과 자동으로 연결되는 구조:

```
[현재]  이상 감지 → RAG 분석 → DB 저장 → 대시보드 표시

[목표]  이상 감지 → RAG 분석 → 챗봇 세션 자동 생성
                              → 운영자가 챗봇에서 추가 질문 가능
                              → "이 이상 해결 후 재실행할까요?" → 승인 연동
```

즉, 단순 알림이 아니라 **AI가 이상을 분석하고, 운영자와 대화하며, 재실행까지 이어주는** 흐름이 최종 목표.

## 9. 테스트 환경 구성

```bash
# 1. 중앙 서버
uv run python scripts/start_server.py --reload

# 2. Mock 에이전트 (정상)
uv run python -m src.agent.main

# 3. Mock 에이전트 (이상 시뮬레이션)
SIMULATE_ANOMALY=true AGENT_DEVICE_ID=MOCK-02 AGENT_PORT=8082 \
  uv run python -m src.agent.main

# 4. 데모 스크립트 실행
uv run python scripts/demo_equipment.py

# 접속
# 대시보드:  http://localhost:8000/equipment
# 챗봇:      http://localhost:8000/chat
# API 문서:  http://localhost:8000/docs
```
